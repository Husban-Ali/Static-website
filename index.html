1. Hard-coded client secret (Insecure)

Storing API keys or secrets directly in frontend code exposes them to anyone viewing the source.

// DON'T DO THIS (demo only)
const INSECURE_API_KEY = "AKIA-EXAMPLE-KEY-12345";
console.log("Using API key:", INSECURE_API_KEY);
Mitigation: Store secrets securely on backend; use environment variables.

2. Insecure contact form (Demo only)

This form simulates a POST request to an insecure endpoint. Data sent over HTTP can be intercepted.

Mitigation: Use HTTPS endpoints and validate inputs on server side.

3. Unsafe DOM manipulation (XSS risk)

Rendering user input using innerHTML without sanitization can cause cross-site scripting (XSS) attacks.

Mitigation: Use textContent or sanitizers like DOMPurify before injecting HTML.

4. Third-party script inclusion (Supply-chain risk)

Including external scripts from untrusted sources can introduce malicious code.

<script src="https://cdn.example.com/widget.js" integrity="..." crossorigin="anonymous"></script>
Mitigation: Use Subresource Integrity (SRI), strict Content Security Policy (CSP), and minimal third-party code.
